## 1. 战略定位重塑：从"AI 学习工具"到"AI 课堂"

### 1.1 当前定位分析

**现状：** 智学是一个 **AI 增强型** 学习平台。AI 生成资源、回答问题、规划路径、追踪进度——但学生是驾驶者，AI 是强大的副驾驶，学生呼唤时才响应。

核心交互模式：
```
学生提问 → AI回答 → 学生阅读 → 学生做题 → AI批改
```
这是一种 **被动响应式** 学习体验。每一步都需要学生主动发起，AI 不会主动说"今天我们来学这个"。

### 1.2 目标定位：AI 个性化课堂

**定位转变：** 从 "AI 学习工具" 到 **"AI 一对一私教课堂"**。

核心交互模式应变为：
```
AI老师: "今天我们来学习梯度消失问题"
       → 讲解 + 黑板画图 + 举例
       → 提问检查理解
学生回答 → AI诊断 → 调整讲解方式
       → 出练习题 → 批改反馈
       → 总结今天所学
       → "明天我们将学习..."
```

**每一天，学生走进这个虚拟课堂，AI 老师已经知道：**
- 昨天学了什么、哪里卡住了
- 今天该学什么、用哪种方式讲
- 学生还有多少时间、精力状态如何

### 1.3 产品原则

| 原则 | 说明 |
|------|------|
| **主动性 > 响应性** | AI 应主动发起教学，而非等待提问 |
| **掌握度 > 进度** | 理解透了再往前走，不是读完就算学会 |
| **个性化是闭环** | 每个交互都反馈到画像，持续优化 |
| **进步可视化** | 让学生每天看到自己的成长 |

### 1.4 竞品定位差异

| vs | 差异点 |
|---|--------|
| Coursera/edX | 非录制课程，而是**实时生成**的个性化内容 |
| ChatGPT | 非通用问答，而是有**课程体系**的教学方法论 |
| Duolingo | 非游戏化操练，而是**深度理解**的 AI 教学 |
| Khan Academy | 非统一内容，而是**千人千面**的实时适配 |

## 2. 个性化学习引擎升级

### 2.1 现状与问题

**当前画像系统（6 维）已经很完善**，但是画像的使用方式比较浅：

- 画像影响的是 **LLM prompt 的措辞**（"请根据这个画像来教学"）
- 而不是改变 **系统的决策逻辑**（"画像显示掌握度低 → 自动插入复习环节"）

**关键缺失：没有真正的掌握度追踪引擎。**

目前掌握度信息分散在各处：
| 数据 | 位置 | 问题 |
|------|------|------|
| knowledge_base (0-1) | StudentProfile JSON 字段 | 由 LLM 模糊估算，无精确算法 |
| quiz_scores | AssessmentRecord JSON | 只存最后一次，无历史轨迹 |
| wrong_answers | localStorage | 前端本地，后端不可见 |

### 2.2 优化建议

#### 2.2.1 概念掌握度追踪引擎（Mastery Engine）

**核心思路：** 建立独立的数据模型 `ConceptMastery`，让每个概念的掌握度有精确的数学计算，而不是依赖 LLM 的模糊估计。

```python
# 新增数据模型
class ConceptMastery(Base):
    user_id: str          # 用户
    concept_id: str       # 概念ID（如 "gradient_descent"）
    mastery_score: float  # 掌握度 0-1
    confidence: float     # 置信度（基于评估次数）
    last_assessed_at: datetime
    assessment_count: int
    forgetting_factor: float  # 遗忘曲线参数
```

**掌握度更新算法：**
```
每次交互（做题/答题/阅读/辅导）→ 更新对应概念的 mastery_score
  - 正确回答: mastery = min(1.0, mastery + 0.15 * (1 - mastery))
  - 错误回答: mastery = max(0.0, mastery - 0.20 * mastery)
  - 长时间未复习: mastery *= (1 - forgetting_factor * elapsed_days)
```

**前端可视化：** 每个概念在学习路径节点上显示实时掌握度进度条，绿色（掌握）→ 黄色（需巩固）→ 红色（需复习）。

#### 2.2.2 先修知识验证系统（Gatekeeper Assessment）

**现状：** 学生可以随意标记任何节点为"已完成"，不验证是否真的掌握了前置知识。

**优化方案：**

1. 每个节点学习前，自动检查前置节点的掌握度
2. 如果前置概念掌握度 < 60%，无法进入新节点
3. 自动生成"补漏迷你课"（Gap-Closing Mini Lesson）
4. 节点标记"已完成"需要满足条件：
   - 所有核心概念的掌握度 ≥ 70%
   - 通过一次简短的章节测验（3-5 题）

```
用户点击 ch04 Transformer → 系统检查前置 ch03 掌握度
  → ch03 掌握度仅 45% → 提示："学习 Transformer 前需要巩固深度学习基础"
  → 自动生成"深度学习基础回顾"迷你课
  → 学生完成回顾 + 通过小测 → 解锁 ch04
```

#### 2.2.3 自适应难度调节引擎

**现状：** 所有内容生成使用相同的 difficulty 参数，不根据学生表现动态调整。

**优化方案：** 引入 **Zone of Proximal Development（最近发展区）** 算法：

```
- 目标：保持学生的答题正确率在 ~70%（太容易则无聊，太难则挫败）
- 每次答题后调整 difficulty_multiplier：
  - 连续 3 题正确 → difficulty += 0.1
  - 连续 2 题错误 → difficulty -= 0.1
  - 每题调整后，重新生成下一题
- 前端显示："当前难度：中等" + 向上/向下箭头示意
```

#### 2.2.4 每日学习计划自动生成

**现状：** 学生打开应用看到 Dashboard 统计数字，但没有"今天该做什么"的结构化指引。

**优化方案：**

```
每天早上（或学生登录时），系统自动生成今日学习计划：

"早上好，小明！你今日有 45 分钟可用时间。
今日计划：
  📖 复习：反向传播（5分钟）——上次正确率60%
  📗 新知识：梯度消失与爆炸（20分钟）
  ✏️ 练习：5 道相关习题（10分钟）
  📊 小测验：章节检测（10分钟）

昨天你花 40 分钟学习了激活函数，测试正确率 85% 👍"
```

**数据来源：**
- 学习路径位置 → 确定今天该学哪个节点
- 掌握度数据 → 确定哪些概念需要复习
- 画像中的可用时间 → 确定学习量
- 遗忘曲线 → 确定复习内容

#### 2.2.5 "今日所学"晚间总结

每天学习结束后，系统自动生成结构化总结：

```
📋 今日学习总结 — 2026-06-23

已掌握：
  ✓ 反向传播算法原理（掌握度 80%）
  ✓ 链式法则求导（掌握度 75%）

需巩固：
  ⚠️ 梯度消失原因（掌握度 45%）
  ⚠️ ReLU vs Sigmoid 选择（掌握度 50%）

明日计划：
  → 继续：梯度消失的解决方案（BatchNorm、ResNet）
  → 复习：前面两个掌握度不足的概念
```

#### 2.2.6 交互历史智能分析

**现状：** conversation_summary 只是对话摘要，不包含学习行为模式分析。

**优化方案：**
- 分析学生的学习时段偏好（早上效率高 vs 晚上头脑清醒）
- 检测内容类型偏好（看代码学得快 vs 读文字理解深）
- 识别挫败模式（哪些话题学生反复出错/放弃）
- 这些分析自动注入到 system prompt 中，指导 AI 教学策略

## 3. 课堂模拟：AI 教师教学体验

### 3.1 核心问题

**现状的问题本质是：没有"上课"的感觉。**

当前的资源页面展示的是一篇 Markdown 文档。你读完了就算学完了。没有老师的声音、没有课堂节奏、没有互动、没有"下课了"的感觉。

### 3.2 优化建议

#### 3.2.1 沉浸式课堂模式（Lecture Mode）

**核心设计：** 当学生进入一个学习节点时，不是展示一个静态 Markdown 页面，而是进入一个全屏的"课堂"。

**课堂结构（每节课 15-30 分钟）：**

```
┌──────────────────────────────────────────┐
│  🎓 课堂中：梯度消失问题         进度 45% │
│──────────────────────────────────────────│
│                                          │
│  ┌────────────────────────────────────┐  │
│  │                                    │  │
│  │   "让我们来看一个简单的三层网络……"  │  │
│  │                                    │  │
│  │   [动画：数据正向传播，梯度逐层减小] │  │
│  │                                    │  │
│  │   "注意到最后一层的梯度是0.001，   │  │
│  │    而第一层的梯度几乎为0……"        │  │
│  │                                    │  │
│  └────────────────────────────────────┘  │
│                                          │
│  ── 我懂了 ── 不太明白 ── 能举例吗？──   │
│                                          │
│  [🤚 举手提问]  [⏸️暂停]  [📝记笔记]     │
└──────────────────────────────────────────┘
```

**每一节课的微结构：**

| 阶段 | 时长 | 交互 |
|------|------|------|
| 导入（Hook） | 1-2 min | AI 抛出一个问题或现象引发好奇心 |
| 讲授（Teach） | 8-12 min | 分段讲解 + 实时图示 + TTS 语音 |
| 示例（Example） | 3-5 min | 具体案例 + 互动式演示 |
| 试做（Try It） | 3-5 min | 学生动手（填空/改代码/选答案） |
| 检查（Check） | 2 min | 快速理解度检查（1-2 题） |
| 总结（Review） | 1-2 min | 回顾要点 + 预告下节 |

**每 2-3 分钟一次苏格拉底式提问：**
```
"为什么这里要用 ReLU 而不是 Sigmoid？思考一下..."
→ 等待 5 秒（给学生思考时间）
→ 如果学生不回答，AI 给出提示
→ 学生回答后，AI 评估并决定下一步
  - 正确 → "很好！我们继续"
  - 错误 → "思路对，但忽略了X，我们再看一次"
```

#### 3.2.2 AI 黑板/白板

**现状：** 知识图谱是静态 DAG，辅导中的 Mermaid 是文字生成的图表。

**优化方案：** AI 在讲解时可以"实时画图"：

```
"想象一个神经元……"
[画一个圆圈 + 输入箭头]

"输入乘以权重后求和……"
[在圆圈内写 Σ

"然后经过激活函数……"
[画出输出箭头 + f() 符号]

"这就像……"
[在旁边画一个类比图]
```

**实现方式：** 扩展 SSE 事件协议，增加 `action` 类型：
```
data: {"type": "action", "action": "draw_circle", "params": {x: 100, y: 200, label: "神经元"}}
data: {"type": "action", "action": "draw_arrow", "params": {from: [50,200], to: [100,200], label: "w*x"}}
data: {"type": "action", "action": "draw_text", "params": {x: 120, y: 220, text: "Σ + f()"}}
```

前端需要一个 Canvas/SVG 白板组件来逐步渲染这些指令。

#### 3.2.3 数字人讲师模式

**现状：** 数字人"诗雅"只用于语音对话，与学习内容完全脱节。

**优化方案：**

- 数字人可作为课堂讲师：
  - 进入课堂模式 → 数字人出现在屏幕角落
  - 讲解时嘴唇同步（TTS + 口型动画）
  - 讲到重点时表情变化（认真/微笑/疑问）
  - 学生回答正确时点头微笑

- 多姿态教学：
  - 讲解时面向"黑板"（内容区）
  - 提问时转向"学生"
  - 听学生回答时做思考状

#### 3.2.4 课堂氛围设计

- 淡雅的课堂背景（非纯白，带柔和纹理或渐变）
- 授课时显示"课时计时器"（还剩 XX 分钟）
- 下课时有"下课铃声"效果或过渡动画
- 连续学习后休息提醒："你已经专注学习了 45 分钟，休息 5 分钟吧"

## 4. 互动式学习体验增强

### 4.1 现状问题

**核心问题：学习是被动的。** 学生阅读或观看内容，不需要任何操作。没有动手环节，没有实验，没有创造。

从认知科学角度看，被动阅读的知识留存率仅 ~10%，而主动学习（讨论/实践/教学）可达 50-90%。

### 4.2 优化建议

#### 4.2.1 交互式代码运行环境

**现状：** 代码示例只是静态高亮文本，学生只能"看代码"不能"跑代码"。

**优化方案：** 集成 Pyodide（浏览器内 Python 运行环境）

```
┌─────────────────────────────────┐
│ # 尝试修改学习率，观察变化       │
│ learning_rate = 0.01  # ← 修改这里│
│ optimizer = SGD(lr=learning_rate)│
│ loss = train_one_epoch()         │
│ print(f"Loss: {loss:.4f}")       │
│                                 │
│ [▶ 运行]  [↺ 重置]              │
│                                 │
│ Loss: 0.4523                    │
│                                 │
│ 💡 试试把学习率改成 0.1，看看   │
│    会发生什么？                 │
└─────────────────────────────────┘
```

**学习效果：** 学生不只是"读代码"，而是**通过实验理解概念**。改一个参数、看结果变化，这是深度学习最有效的方式之一。

#### 4.2.2 交互式可视化

让概念"动起来"：

- **前向传播动画：** 数据从输入层流向输出层的实时动画，每个节点的激活值以颜色深浅表示
- **决策边界演示：** 学生拖拽数据点，观察分类边界如何变化
- **损失函数地形图：** 展示不同优化器（SGD/Adam）在损失面上的下降路径
- **注意力热力图：** 可视化 Transformer 中各 token 的注意力权重分布

**实现方式：** 前端 Canvas/SVG 组件 + LLM 生成的配置数据（非实时渲染引擎，而是生成结构化的可视化描述数据）。

#### 4.2.3 引导式笔记（Guided Note-Taking）

**现状：** 没有笔记功能，学生学完就忘。

**优化方案：** AI 在讲解时故意留下"空白"：

```
"梯度下降的核心思想是：沿着____的方向更新参数。"
→ 学生输入答案
→ 自动批改
→ 正确 → "对的！是损失函数下降的方向"
→ 错误 → "接近了，再想想... 损失函数是变大还是变小？"
```

每节课后自动生成一份**结构化笔记**，包含：
- 本节课的核心概念列表
- AI 生成的要点总结
- 学生填写的空白答案
- 个性化记忆口诀（AI 根据学生易错点生成）

#### 4.2.4 概念图构建（学生自己画）

**现状：** 知识图谱是系统生成的 DAG，学生只是查看。

**优化方案：** 每学完一课/一章，让学生自己构建概念图：

```
"请把本节课学的概念连接起来"
[画布]
  - 反向传播
  - 链式法则
  - 梯度
  - 损失函数

→ 学生拖拽连线：反向传播 → 链式法则，梯度 → 反向传播
→ 系统与"专家图"对比
→ "你漏掉了损失函数 → 梯度 这条连接"
→ 自动补课："我们来复习一下损失函数和梯度的关系"
```

这是一种高认知负荷的主动学习方式，**效果远超阅读和选择题**。

#### 4.2.5 多模态内容增强

**现状：** 只有文字 + Mermaid 图表。没有图像、没有音频、没有视频。

**优化方案：**

1. **AI 生成插图：** 对关键概念，让 LLM 生成图表描述（SVG 代码或绘图指令）
   - 如"卷积操作示意图"、"Transformer 架构图"
   - 用纯前端 SVG 渲染，无需外部 API

2. **语音朗读：** 所有文字内容附加 TTS 朗读按钮
   - 适合通勤/睡前/休息时的"听学"模式
   - 前端 Web Speech API 即可实现，无需额外服务

3. **视频化课程：** 幻灯片 + 语音旁白 + 动画标注的自动合成
   - 当前生成的一篇"文档"可一键切换为"视频模式"
   - 翻页 + 语音 + 高亮，模拟短视频课程

#### 4.2.6 学习工具箱

**现状：** 没有持久化的学习工具。

**新增功能：**

| 工具 | 说明 |
|------|------|
| **荧光笔标注** | 学生可高亮资源中的重要段落，高亮内容自动进入复习卡 |
| **个人术语表** | 难词/专业术语自动收录，AI 生成个性化助记法 |
| **书签系统** | 标记重要位置，支持分类和备注 |
| **闪卡系统** | 每个核心概念 AI 自动生成 Q&A 闪卡，加入间隔复习队列 |
| **思考提示** | 学完后 AI 提示："用你自己的话解释过拟合"——学生输入后 AI 评估 |

#### 4.2.7 联网搜索增强互动

**现状：** 联网搜索只在辅导/资源生成时后台使用。

**优化方案：** 课堂中 AI 可动态搜索实时资料：

```
"Transformer 在 2025 年之后有哪些新进展？让我查一下……"
[联网搜索 → 实时集成到讲解中]
"根据最新论文，Mamba 架构在某些任务上已经可以替代 Transformer……"
```

使课程内容**永远不过时**，AI 实时融合最新知识。

## 5. 评估与反馈体系重构

### 5.1 现状问题

**评估层面的关键缺陷：**

1. **题型单一：** 只有四选一选择题 → 无法评估理解深度
2. **反馈浅层：** 对/错 + 静态解析 → 无诊断性分析
3. **无代码评估：** AI 课程的核心（编程）完全没有评估
4. **间隔复习薄弱：** SM-2 只用于错题，没有概念闪卡系统
5. **画像更新滞后：** LLM 模糊评估掌握度，不精确

### 5.2 优化建议

#### 5.2.1 题型扩展

**新增题型矩阵：**

| 题型 | 评估维度 | 批改方式 | 难度 |
|------|----------|----------|------|
| 选择题 | 知识记忆 + 辨别 | 自动批改 | 低 |
| 填空题 | 精确回忆 | 关键词匹配 | 中 |
| 简答题 | 理解程度 + 表达 | AI 语义评分 | 高 |
| 代码题 | 编程 + 理解 | 测试用例 + 静态分析 | 高 |
| 场景题 | 应用 + 推理 | AI 评估推理链 | 高 |
| 口试题 | 即时反应 + 表达 | TTS + STT + AI 评分 | 高 |

**简答题评分示例：**
```
问题："用自己的话解释注意力机制"

学生A回答："注意力机制是让模型关注输入中的重要部分"
→ AI 评分：75/100
→ 诊断：理解正确但缺乏细节
→ 建议："能否具体说说 QKV 分别代表什么？"

学生B回答："通过计算查询和键的相似度来加权值"
→ AI 评分：92/100
→ 诊断：理解准确，术语使用正确
→ 鼓励："非常精确！"
```

#### 5.2.2 诊断性错误分析

**现状：** 批改只告诉学生对错，不告诉错的类型。

**优化方案：** 每个错误答案进行**错误类型诊断**：

| 错误类型 | 特征 | 补救策略 |
|----------|------|----------|
| **概念混淆** | 选了相近但错误的概念 | 对比讲解两个概念的区别 |
| **知识遗漏** | 没学过这个概念 | 回溯到前置知识点补课 |
| **粗心失误** | 其他都对，就这题错 | 轻松提醒，继续前进 |
| **推理偏差** | 思路方向对但细节错 | 引导式追问，让学生自己发现错误 |

**交互示例：**
```
学生答错："Batch Normalization 的作用是防止过拟合"
→ AI 诊断：
  "你的回答方向接近了，但 BatchNorm 主要是解决内部协变量偏移
  和梯度消失问题。你想一下，它和 Dropout 的作用有什么不同？"
→ 追问："那 Dropout 是用来做什么的？"
→ 确认学生确实区分了这两个概念
```

#### 5.2.3 代码评估系统

**现状：** 代码是"教学内容"而非"评估工具"。

**优化方案：** 在 Quiz 页面新增代码题模式：

```
题目：实现一个函数，计算二元交叉熵损失

def binary_cross_entropy(y_true, y_pred):
    # 请在这里编写代码
    _______

[▶ 运行] [💡 提示] [✅ 提交]

后端评估：
  1. 测试用例通过率（70%）
  2. 代码规范/效率（15%）
  3. 边界情况处理（15%）

AI 反馈：
  "你的实现是正确的！但有两个改进点：
  1. 没有处理 y_pred 为 0 的情况（会导致 log(0)）
  2. 可以用 np.clip 来数值稳定化
  试试加一个很小的 epsilon：y_pred = np.clip(y_pred, 1e-7, 1-1e-7)"
```

#### 5.2.4 概念闪卡 + 间隔复习系统

**现状：** 间隔复习只针对错题，没有针对概念的主动记忆系统。

**优化方案：** 每节课后 AI 自动生成概念闪卡：

```
"本节课生成了 8 张闪卡，已加入你的复习队列"

📇 今日复习（5 张待复习）：
  1. [反向传播] 什么是链式法则？ → 预计 30s
  2. [梯度消失] 为什么深层网络梯度会消失？ → 预计 45s
  3. [ReLU] 为什么 ReLU 能缓解梯度消失？ → 预计 30s
  ...

[▶ 开始今日复习] 预计耗时 4 分钟

复习时：
  - 正面显示概念名 → 学生回忆
  - 点击翻转 → 显示 AI 生成的答案
  - 自评：记得 / 模糊 / 忘了
  - 根据自评调整下一次复习间隔（SM-2 算法）
```

#### 5.2.5 信心校准（Confidence Calibration）

**现状：** 无法判断学生是否"以为自己懂了但其实没懂"。

**优化方案：** 每道题回答后，让学生评估自己的信心：

```
"你对这个答案有多少把握？"
  🔵 非常确定（100%）
  🟢 比较确定（75%）
  🟡 一般（50%）
  🟠 不太确定（25%）
  🔴 猜的（0%）

→ 系统追踪"校准曲线"：
  当你"非常确定"时，正确率是 90% ——说明你的自我认知很准确
  当你"比较确定"时，只有 40% 正确率 ——你有点过度自信了
  "建议：在这几个知识点上，你实际理解没有自己想的那么深，建议复习：……"
```

**这是元认知训练**，帮助学生更好地评估自己的知识边界。

## 6. 动力机制与社交学习

### 6.1 现状问题

**当前的激励系统（8 个成就 + 连续打卡）是一个良好的起点，但存在不足：**

1. **成就感即时性不足：** 成就需要长时间积累，日常缺少"小胜利"
2. **无短期目标：** 没有"今天完成什么"的明确指引
3. **完全孤立：** 没有社交元素，学习是孤独的
4. **无角色成长：** 没有等级/段位/成长感

### 6.2 优化建议

#### 6.2.1 每日任务系统

```
📋 今日任务（完成 3/3 可获得 50 XP）

  [✅] 完成一次 10 分钟以上的学习  +20 XP
  [✅] 正确回答 5 道练习题       +20 XP
  [  ] 复习 3 张闪卡              +15 XP

  额外挑战：连续 3 天完成所有任务  +50 XP 🏆
```

**任务设计原则：**
- 难度适中：每天 ~20 分钟可完成全部
- 多样性：阅读、做题、复习、创造各类型
- 递进性：连续完成有额外奖励

#### 6.2.2 学习合约系统

```
📝 学习合约：本周承诺

"我，小明，承诺本周（6/23-6/29）：
  ✅ 每天学习至少 30 分钟
  ✅ 完成"深度学习"章节
  ✅ 周末前正确率保持在 70% 以上

  如果完成：解锁限定徽章 🏅
  如果未完成：AGI 小助手会发来鼓励 💪"
```

**心理学原理：** 公开承诺提升执行力。AI 作为"见证人"增加责任感。

#### 6.2.3 等级成长系统

**现状：** 8 个成就是离散的，没有连续的成长感。

**优化方案：** 引入等级体系（类似 RPG）：

| 等级 | 称号 | 升级条件 | 解锁 |
|------|------|----------|------|
| Lv.1 | 初学者 | 注册 | 基础功能 |
| Lv.5 | 探索者 | 完成 1 章 | 笔记功能 |
| Lv.10 | 学徒 | 掌握 20 个概念 | 挑战模式 |
| Lv.20 | 学者 | 通过 5 次测评 | AI 助教权限 |
| Lv.30 | 专家 | 掌握 50 个概念 | 自定义学习路径 |
| Lv.50 | 大师 | 完成全部 10 章 | 🏆 终极成就 |

**经验值（XP）来源多样化：**
- 阅读资源：+2 XP/min
- 做对题目：+10 XP/题
- 连续打卡：+5 XP/天 × 连续天数
- 完成每日任务：+50 XP
- 创建笔记：+15 XP/条
- 教会别人（写解释）：+30 XP/次

#### 6.2.4 谨慎的社交学习

**现状：** 完全没有社交功能。

**注意：** 社交功能是一把双刃剑，做不好会带来攀比、焦虑、隐私问题。建议以**匿名 + 辅助**的方式引入。

**方案 A：同侪比较（匿名，可选加入）**
```
本周学习时长对比（您在第 4 组相似水平学生中）：

  组内平均：4.2 小时/周
  您的：6.5 小时/周  ↑54%

  这个概念上，您比 80% 的同学掌握得更好 👍
```

**方案 B：AI 引导的小组讨论**
```
AI： "我们来自动组成一个 4 人学习小组。
  请大家用自己的话解释什么是过拟合。"

  同学A："模型死记硬背了训练数据……"
  同学B："在训练集上表现很好但在新数据上表现很差……"

  AI："很好！请每位同学点评一下其他人的回答，
    指出一个值得学习的点。"
```

**方案 C：知识分享与互教**
```
"你刚刚掌握了『注意力机制』，你的掌握度评分很高。
  想试试写一段解释帮助其他同学吗？
  好的解释可以获得 '知识之星' 徽章 🌟"
```

#### 7.2.3 课堂状态机（前端状态管理）

**现状：** 前端页面是静态的，没有"学习模式"概念。

**优化：** 引入学习会话状态机：

```
状态流转：
    IDLE
      ↓ (点击学习节点)
    WARMUP ─→ LECTURE ─→ PRACTICE ─→ ASSESS ─→ REVIEW ─→ COMPLETE
      ↑          ↑           ↑          ↑          ↑
      │          │           │          │          │
      └──────────┴───────────┴──────────┴──────────┘
                     (任何时候可退出，下次从检查点恢复)

每个状态对应不同的 UI 布局：
  - IDLE: 当前 Dashboard
  - WARMUP: 快速回顾前置知识（3-5 题闪卡）
  - LECTURE: 全屏沉浸式课堂（数字人/黑板/分段讲解）
  - PRACTICE: 交互式练习区（代码/填空/选择）
  - ASSESS: 章节检测（综合题型）
  - REVIEW: 课堂总结 + 生成笔记 + 闪卡
  - COMPLETE: 完成动画 + 今日总结
```

**Store 设计：**

```typescript
interface LearningSession {

  currentNodeId: string;

  totalSteps: number;         // 总步骤数
  lessonContent: LessonStep[];
  interactionHistory: Interaction[];
  sessionStartTime: number;
  masteryBefore: ConceptMastery[];
  masteryAfter: ConceptMastery[];
}
```

#### 7.2.5 内容生成管线升级

**现状：** 单次 LLM 调用一次性输出完整内容。

**优化方案——两阶段流水线（参照 OpenMAIC 模式）：**

```
第一阶段：大纲生成
  [用户输入 + 画像 + 知识图谱]
  → LLM 生成：结构化大纲
    {
      "title": "梯度消失问题",
      "estimated_duration": "25分钟",
      "sections": [
        {"id": "s1", "title": "什么是梯度消失", "type": "lecture", "estimated_minutes": 8},
        {"id": "s2", "title": "为什么会出现", "type": "lecture", "estimated_minutes": 7},
        {"id": "s3", "title": "代码演示", "type": "code", "estimated_minutes": 5},
        {"id": "s4", "title": "练习巩固", "type": "practice", "estimated_minutes": 5}
      ]
    }

第二阶段：逐项内容生成
  → 按大纲逐段生成完整内容
  → 学生每学完一段，异步生成下一段
  → 支持断点续传、部分重生成
  → 可缓存大纲，下次复用
```

**优势：**
1. 学生看到大纲后可以调整（跳过/深入）
2. 内容生成和呈现可以**流式并行**
3. 每段内容独立，AI 生成质量更高
4. 教师/管理员可审核大纲
5. 缓存机制（相同大纲不重复生成）

#### 7.2.6 持久化存储优化

**现状问题：**
- PPT 任务状态存在内存中（服务器重启丢失）
- 错题存在 localStorage（跨设备丢失）
- 学习行为记录粒度不够

**优化：**
- PPT 任务 → SQLite 持久化 + 清理策略
- 错题本 → 后端 API + DB 持久化
- 学习行为 → Interaction Log 表（支持分析）

## 附：核心认知

### 最重要的一个改变

> **当前系统是"学生找知识"：** 学生打开页面、选择功能、发起请求、等待 AI 响应。
>
> **理想系统是"知识找学生"：** 学生登录 → AI 已经准备好了今天的学习计划 → 主动开始教学 → 在需要时提问 → 在结束时总结。

**从 Dashboard（仪表盘）到 Classroom（课堂），是这个系统最本质的进化。**

### 三个不能丢的原则

1. **个性化不是形容词，是引擎：** 画像不是展示给用户看的"资料卡"，而是驱动所有决策的算法输入。
2. **内容不是最终产出，学习才是：** 生成一篇漂亮的 Markdown 不是终点，学生理解了这个概念才是。
3. **AI 不是工具，是老师：** 学生需要的不是一个更强的搜索引擎，而是一个知道"该怎么教你"的智能导师。
