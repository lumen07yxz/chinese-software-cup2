# 第九章：MLOps 与 AI 工程实践

## 9.1 MLOps 概述

### 9.1.1 为什么需要 MLOps

MLOps（Machine Learning Operations）是 ML 与 DevOps 的交叉学科，旨在统一 ML 系统的开发和运维。它解决了以下几个根本性的行业痛点：

1. **传统软件与ML系统的差异**：
   - 传统系统有确定的逻辑（输入→输出），ML系统有不确定性（训练数据和模型性能随时间退化）
   - 传统系统测试可单元化，ML系统需要验证数据、训练流程和模型行为的正确性

2. **技术债务累积更快**（Sculley et al., 2015）：
   - **纠缠（Entanglement）**：ML系统中几乎不可能对某个特征/模型行为做独立改变的测试，所有改变相互耦合
   - **隐藏反馈循环**：模型输出影响用户行为→用户数据送入下一轮训练→造成真实数据分布漂移
   - **不可见的监控债务**：依赖非结构化日志和回溯分析而非主动监控

3. **模型性能退化**：数据分布漂移和概念漂移持续发生，模型需要持续监控和重新训练

### 9.1.2 MLOps 成熟度模型

| 级别 | 特征 | 典型工具 | 组织规模适用情景 |
|------|------|----------|----------------|
| Level 0 - 手动 | 无CI/CD、手动训练、手动部署 | Jupyter Notebook | 个人研究/PoC |
| Level 1 - 半自动 | CI/CD自动化训练+部署，手动特征/模型管理 | MLflow、Airflow | 中小团队 |
| Level 2 - 全自动 | 自动化CI/CD/CT，特征引擎，模型监控 | MLflow+Kubeflow+Evidently | 成熟ML团队 |

## 9.2 ML 项目生命周期

### 9.2.1 数据管理

**数据版本控制**：
- **DVC**（Data Version Control）：类Git命令操作大文件/数据集版本，与Git Repo解耦
- **LakeFS**：基于对象存储的数据仓库层，支持Git-like的分支/合并语义操作数据
- **Delta Lake**：ACID事务 + 时间旅行（time travel，回溯到历史版本）

**特征存储（Feature Store）**：
- 统一管理在线/离线特征（Offline = 用于训练的历史快照，Online = 用于推理的最新特征）
- **Feast**：开源特征存储，连接到离线存储（S3/GCS/Parquet）+ 在线存储（Redis/DynamoDB）+ 时间点查询功能
- **Tecton**：商业版，自动处理时间点正确性（point-in-time correctness——确保训练特征不"看到未来"）

**数据质量监控**：
$$质量指标: \text{完整性} (非空比例)、\text{唯一性} (去重比率)、\text{一致性} (分桶间统计一致性)、\text{时效性} (数据延迟最大可接受时间)$$

### 9.2.2 实验追踪

| 平台 | 跟踪内容 | 优势 | 劣势 |
|------|---------|------|------|
| **MLflow** | 参数/指标/模型/代码 | 开源、轻量、通用 | 分布式协调有限 |
| **Weights & Biases** | 完整的实验跟踪+超参数搜索+报告 | UI最好、协作强大 | 云服务（有离线版） |
| **TensorBoard** | 深度学习训练指标/可视化 | TF集成最佳 | 非TF项目不太友好 |
| **Aim** | 开源轻量（与MLflow功能相似但UI更好） | 开源UI设计好 | 生态小 |

**追踪内容的黄金标准**：
- ✅ 超参数（网络结构、学习率、batch、优化器设置）
- ✅ 代码/数据集版本号（git commit hash + data hash）
- ✅ 评价指标（所有下游自定义指标 + loss曲线）
- ✅ 模型文件（权重——断点续训 / 注册部署）
- ✅ 训练环境（package version、OS、GPU型号）

### 9.2.3 模型注册与版本管理（MLflow Model Registry）

```mermaid

    A[训练脚本] --> B[日志记录到Tracking]
    B --> C[注册模型]
    C --> D[Stage: Staging]
    D --> E[Stage: Production]
    E --> F[Stage: Archived]

```

每个模型版本在注册表中的生命周期：
1. **None（未指定）**：刚注册，未分配阶段
2. **Staging**：通过了CI/CD测试，准备部署前验证
3. **Production**：已上线，提供生产推理服务
4. **Archived**：被新版本替代，但仍可回溯

## 9.3 模型部署

### 9.3.1 部署模式比较

| 模式 | 延迟要求 | 吞吐量 | 典型框架 | 适用场景 |
|------|---------|--------|---------|---------|
| **批处理（Batch）** | 小时级 | 高 | Spark批推理、Airflow | 报表/推荐/风控 |
| **实时API（REST/gRPC）** | 毫秒-秒级 | 中-高 | FastAPI、Triton、TorchServe | 在线推荐/搜索/对话 |
| **边缘/设备端** | 实时（<10ms） | 低 | TensorFlow Lite、ONNX Runtime | IoT/移动/车载 |
| **流处理** | 近实时（秒级） | 高 | Kafka Streams、Flink | 异常检测/事件预测 |

### 9.3.2 模型服务框架

- **Triton Inference Server（NVIDIA）**：支持PyTorch/TF/ONNX/TensorRT等多框架并发推理，动态批处理、模型并发、GPU共享
- **TorchServe**：PyTorch原生服务框架，支持多模型加载/版本管理/REST API/gRPC
- **vLLM**：大语言模型高性能推理——Continuous Batching（无需等待单个请求完成就为新请求分配空位）+ PagedAttention（类似分页内存管理KV Cache）→ 吞吐量提升8-20x
- **Ollama**：本地LLM推理的简化部署工具
- **Ray Serve**：分布式模型服务，天然支持Python、可将AI Pipeline分解为多个阶段（数据预处理 → 模型调用 → 后处理）

### 9.3.3 模型优化关键技术

| 技术 | 原理 | 效果（典型值） | 适用阶段 |
|------|------|---------------|---------|
| **量化（Quantization）** | fp32 → int8/int4 | 模型大小75%↓，吞吐量2-4x | 训练后 |
| **剪枝（Pruning）** | 移除小绝对值权重 | 模型大小50%↓，精度略微下降 | 训练后 |
| **蒸馏（Distillation）** | 小模型学习大模型输出分布 | 在保持体积小~90%的同时质量仍可接受 | 训练时 |
| **结构优化（TensorRT/OpenVINO）** | 计算图优化（层融合/常数折叠/Kernel自动调优） | 2-5x加速 | 训练后 |

## 9.4 模型监控

### 9.4.1 监控指标体系

| 监控类型 | 指标名称 | 检测方式 | 危害等级 |
|---------|---------|---------|---------|
| **数据漂移（Data Drift）** | 特征分布的KL散度/PSI（总体稳定性指数） | 比较参考分布与当前分布 | 高 |
| **概念漂移（Concept Drift）** | 模型预测准确率 / log-loss / 残差 | 同时监控输出分布+ground truth延迟 | 很高 |
| **异常检测** | 特征值离群点率 / 速度变化 | z-score / 孤立森林 | 中 |
| **服务指标** | 延迟P50/P99 / 吞吐量 / 错误率 | 常规APM（如Grafana） | 中 |

**PSI（稳定性指数，Stability Index，用于检测分布漂移的计算标准）**：
$$PSI = \sum_{i=1}^n (p_i - q_i) \cdot \ln\frac{p_i}{q_i}$$

PSI<0.1——分布稳定，0.1-0.25——需关注，>0.25——严重漂移

## 9.5 CI/CD for ML

### 9.5.1 CI（持续集成）的ML特有问题

- **数据验证**：schema检查（字段类型、非空率、值范围一致性）
- **模型训练测试**：在小采样数据上完整训练一次（验证代码逻辑无bug）
- **模型评估**：在保留测试集上的评估指标检查——如果新模型相对当前生产模型的退化超过阈值则不应通过
- **模型可复现**：训练环境及数据hash绑定，确保其他人也能复现结果

### 9.5.2 CD（持续部署）策略

- **A/B测试**：新老模型在线流量分流，对比业务指标
- **金丝雀部署（Canary）**：新模型首先接收一小部分流量（如5%），确认无问题后逐步扩大
- **蓝绿部署**：全量新版本并列运行对应整组服务→流量切换→快速回滚
- **自动回滚**：如果部署后监控指标（延迟、错误率、业务指标）超过阈值→自动下线新版本，恢复到稳定版本

## 9.6 大模型的工程挑战

| 挑战 | 方法 | 说明 |
|------|------|------|
| **推理成本** | **KV Cache** | 存储已生成的Key-Value，避免重复计算 |
| **长上下文** | **Ring Attention** / **StreamingLLM** | 分布式多GPU切分注意力、只保留关键位置缓存 |
| **推理加速** | **投机解码 (Speculative Decoding)** | 用小型"草稿"模型生成候选，大模型快速验证—大模型步数从tokens*O(n)→~O(k)步验证 |
| **并发服务** | **Continuous Batching** | 动态批处理（vLLM的核心）：每个请求自独立，服务不必等一个请求完成就为新请求分配空闲slot |
| **KV Cache过大** | **PagedAttention** | 虚拟内存式KV Cache分页管理，类似操作系统的分页机制 |
| **微调效率** | **LoRA / QLoRA** | 量化为4-bit + 低秩适配器训练 |
